# Conversational Memory Recall ("Deja Vu") Design

**Date:** 2026-02-23
**Status:** Approved

## Problem

Friday has no way to recall information from past conversations. When the user says "remember when we discussed..." or "we talked about this before", Friday has zero access to her own conversation history during a live session. This makes her feel amnesiac despite having all past conversations stored in SQLite.

## Solution

A tool-based recall system backed by FTS5 full-text search on conversation summaries. The LLM autonomously decides when to search memory — triggered by user cues like "do you remember" or sensing missing context. Two-step pattern: search summaries first (skim), then drill into full messages (deep read).

MCU codename: **Deja Vu** — Friday reaching into her own memory.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Trigger mechanism | Tool-based (LLM decides) | Most natural — Friday *chooses* to think back. No wasted search on every message. Avoids prompt bloat from always-on enrichment. |
| Search target | FTS5 on summaries | Summaries are 1-3 sentences, already generated by ConversationSummarizer. Small index, fast search, good semantic coverage. |
| Drill-down | Full message retrieval by session ID | Two-step recall mirrors human memory: fuzzy recognition → specific details. |
| Backfill | On first boot | Existing conversations with summaries get indexed immediately. Full memory reach from day one. |
| Index namespace | `"conversations"` in existing `embeddings_fts` | Reuses the proven embed/search/forget infrastructure. No new tables. |

## Architecture

```
User says "remember when we..."
  → LLM decides to call recall_memory(query: "...", mode: "search")
  → FTS5 search against conversation summaries (namespace: "conversations")
  → Returns ranked list: session ID, date, summary text
  → LLM reads results, optionally calls recall_memory(sessionId: "abc", mode: "recall")
  → Returns full message history for that session (truncated)
  → LLM weaves recalled context into response naturally
```

### FTS5 Indexing Lifecycle

**Index at save-time:**
- `saveConversation()` → if summary exists, call `indexConversation(session)`
- `indexConversation()` embeds summary text under namespace `"conversations"` with metadata `{ sessionId, date }`
- Embedding ID stored in KV (`"conversations"/<sessionId>` → embedding ID) for later cleanup

**Backfill on boot:**
- `FridayRuntime.boot()` checks KV flag `"conversations"/"backfill-done"`
- If absent: scan all existing conversations with summaries, batch-embed, set flag
- One-time operation, idempotent

**Prune coordination:**
- When `saveConversation()` prunes sessions beyond the 500-cap, also forget their FTS5 embeddings
- Look up embedding ID from KV metadata, call `forget("conversations", embeddingId)`

### The recall_memory Tool

**File:** `src/core/recall-tool.ts`

Factory function pattern (like `createEnvironmentTool`):

```typescript
createRecallTool(memory: SQLiteMemory): FridayTool
```

**Parameters:**

| Param | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| query | string | Yes (search) | — | FTS5 search terms |
| mode | string | No | "search" | `"search"` or `"recall"` |
| sessionId | string | Yes (recall) | — | Session to retrieve full messages from |
| limit | number | No | 5 | Max results for search mode |

**Clearance:** `[]` — reading own memory is non-privileged.

**Registration:** `runtime.ts`, after memory init, alongside sensorium tool.

### Output Formats

**Search mode:**
```
Found 3 matching conversations:

1. [2026-02-22 14:30] (session abc-123)
   "Discussed Docker networking, bridge networks, and DNS resolution."

2. [2026-02-21 09:15] (session def-456)
   "Implemented SMARTS knowledge system with FTS5 indexing."
```

**Recall mode:**
- First 50 messages of the session
- Each message truncated to 500 characters
- Total output capped at ~8000 characters
- Format: `user: <text>\nassistant: <text>`

### System Prompt Guidance

Added to `SYSTEM_PROMPT` in `prompts.ts`:

> When the user references a past conversation ("remember when...", "we discussed...", "last time..."), or when you sense missing context that a previous session might have covered, use the `recall_memory` tool to search your conversation history. First search with relevant keywords, then recall specific sessions for details. Think of this as reaching into your memory — acknowledge what you're doing naturally ("Let me think back... yes, I remember that conversation.").

## Testing Strategy

- SQLiteMemory: indexConversation, searchConversations, backfill idempotency, prune cleanup
- Recall tool: search found/not-found, recall found/not-found, missing params, error handling
- Integration: save → index → search → verify match chain

## What This Does NOT Include

- Automatic system prompt enrichment (rejected — tool-based trigger is cleaner)
- Full message text indexing (only summaries — keeps index small and relevant)
- `/history search` protocol command (can be added later if useful)
- Cross-session context threading (future — linking related conversations)
